saisoku / GBF Master Data Operation

GBFマスタデータ運用手順

キャラ・武器・召喚石・ジョブ・素材・クエストのマスタデータを、安全に追加・更新するための運用ガイドです。 DBマスタを優先しつつ、fallbackと既存投稿互換を維持する方針をまとめています。

DB優先 + fallback DB取得に失敗しても静的マスタで最低限動作します。
name保存を維持 既存投稿互換を保ちながら masterId を併存します。
thumbnailPath運用 画像は gbf-asset bucket の相対パスで管理します。

1目的

この文書は、キャラ・武器・召喚石・ジョブ・素材・クエストのマスタデータを安全に追加・更新するための運用手順です。

現在の基本方針 saisoku では、投稿データの name 保存を維持しつつ、キャラ・武器・召喚石の detail JSON に masterId を併存保存します。マスタはDBを優先し、取得できない場合は src/lib/buildMasters.ts の fallback を使います。

2管理場所

DB投入用 seed は以下で管理します。

server/data/gbfMasterSeed/

現在の構成

ファイル 役割
types.tsseedデータ用の型定義
jobs.tsジョブマスタ
characters.tsキャラマスタ
summons.ts召喚石マスタ
weapons.ts武器マスタ
index.ts各種別のseedを結合して gbfMasterSeedItems として公開
投入方法 prisma/seed.ts は server/data/gbfMasterSeed/index.ts から gbfMasterSeedItems を読み込み、upsert でDBに投入します。

3kind の種類

GbfMasterKind は以下を想定します。

character weapon summon job material quest
現時点の投稿フォーム候補で使うのは character / weapon / summon / job です。素材・クエストは今後の追加対象ですが、 本格投入や relation 設計は別フェーズで行います。

4id 命名ルール

id は投稿 detail JSON の masterId として保存されるため、原則として後から変更しません。

推奨形式

character: char-<slug>
weapon: weapon-<slug>
summon: summon-<slug>
job: job-<slug>
material: material-<slug>
quest: quest-<slug>

例

char-haaselia
weapon-dark-opus
summon-varuna
job-yamato
material-example
quest-mugen-hl
  • ✓ASCII lowercase と - を使う
  • ✓日本語名をそのまま id にしない
  • ✓同じ種類内で重複させない
  • ✓既存 id は変更しない
  • ✓表記ゆれや略称は aliases に入れる
注意 既存IDに改善余地があっても、投稿済みデータとの互換性を優先し、変更せず別途課題として扱います。

5thumbnailPath ルール

Supabase Storage bucket は以下を前提にします。

gbf-asset

thumbnailPath は bucket より後ろの相対パスだけを書きます。seed にフルURLは基本的に書きません。

推奨形式

characters/<slug>.webp
weapons/<slug>.webp
summons/<slug>.webp
jobs/<slug>.webp
materials/<slug>.webp
quests/<slug>.webp

例

characters/haaselia.webp
weapons/dark-opus.webp
summons/varuna.webp
jobs/yamato.webp

画像がある場合

thumbnailPath に相対パスを書きます。

thumbnailPath: "characters/haaselia.webp"

画像が未準備の場合

thumbnailPath を省略します。UIはプレースホルダー表示に fallback します。

// thumbnailPath なし
thumbnailUrl は例外的な直接URL用に残しますが、通常運用では thumbnailPath を使います。

6aliases の書き方

aliases は検索・表記ゆれ吸収用です。正式名称そのものを大量に重複させる必要はありません。 

aliases: ["ハーゼ", "ハーゼリーラ"]
  • ✓略称、俗称、表記ゆれを入れる
  • ✓空文字を入れない
  • ✓同じ item 内で重複させない
  • ✓別 item と紛らわしい alias は避ける
seed 実行時、alias は normalizedAlias に正規化され、 GbfMasterAlias に upsert されます。

7tags の書き方

tags は候補表示・絞り込み・簡易分類に使います。

tags: ["水", "十賢者", "高難度"]
tags: ["リミテッド", "周回"]
tags: ["神石", "メイン"]
  • ✓UIで見せても自然な短い語にする
  • ✓属性、シリーズ、用途、カテゴリを入れる
  • ✓細かすぎる内部情報は metadata に入れる
  • ✓外部サイト由来の説明文や攻略文を丸ごと入れない

8metadata の書き方

metadata は kind ごとの差分項目を入れる場所です。何でも入れすぎず、検索や表示に使う項目を中心にします。

武器

metadata: {
  weaponType: "刀",
  series: "オメガ"
}

キャラ

metadata: {
  role: "support"
}

クエスト

metadata: {
  questGroup: "レヴァンス",
  difficulty: "HL",
  contentType: "周回・武器集め用"
}

素材

metadata: {
  materialCategory: "強化素材"
}
  • ✓キー名は英語の camelCase にする
  • ✓値はできるだけ string / number / boolean / string[] に留める
  • ✓relation 的な情報は、将来 relation テーブルを追加するまで最小限にする

9追加手順

  1. 対象 kind の seed ファイルを開く。 
    キャラ: server/data/gbfMasterSeed/characters.ts
武器: server/data/gbfMasterSeed/weapons.ts
召喚石: server/data/gbfMasterSeed/summons.ts
ジョブ: server/data/gbfMasterSeed/jobs.ts
  2. GbfMasterSeedItem を追加する。
  3. id が命名ルールに合っているか確認する。
  4. 画像がある場合は thumbnailPath を相対パスで書く。
  5. 必要に応じて aliases / tags / metadata を追加する。
  6. seed を実行する。 
    npm run prisma:seed
  7. 型チェックとビルドを実行する。 
    npm run typecheck
npm run build
  8. ログイン済み状態で GET /api/build-masters を確認する。

確認観点

  • ✓itemCount が想定件数になっている
  • ✓既存33件が消えていない
  • ✓追加した item の id / kind / name が正しい
  • ✓alias を追加した場合は aliases に返ってくる
  • ✓投稿フォーム候補表示が空になっていない

10seed の注意点

npm run prisma:seed は upsert です。既存 id の item は更新され、新しい id は追加されます。

注意 seed から削除しても、現在の seed 処理ではDB上の既存 item を自動削除しません。使わなくなった item は、削除ではなく isActive 運用を検討します。
  • ✓id を変えると別 item として追加される
  • ✓既存 id の変更は、投稿済み masterId との互換性を壊す可能性がある
  • ✓seed から削除してもDB上の既存 item は自動削除されない
  • ✓使わなくなった item は isActive 運用を検討する
  • ✓フルURLを直接持たせるより thumbnailPath を優先する
