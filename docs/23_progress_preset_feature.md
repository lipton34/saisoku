## 1. この文書の位置づけ

この文書は、「必要数の進捗」と素材収集プリセット機能の仕様、優先順位、計算方法、互換性、開発手順を一か所にまとめるための機能別仕様書である。

この機能を調査、設計、実装、修正、レビューする場合は、必ず以下の順に確認する。

1. `AGENTS.md`
2. `docs/01_overall_policy.md`
3. 本文書 `docs/23_progress_preset_feature.md`
4. 素材数と出典の資料 `docs/preset.md`
5. 現行コード、Prisma schema、migration、Git履歴

役割分担は次のとおり。

| 文書 | 役割 |
| --- | --- |
| `docs/01_overall_policy.md` | Saisoku全体のプロダクト方針 |
| 本文書 | 本機能の振る舞い、優先順位、データ構造、実装・運用ルール |
| `docs/preset.md` | 段階別素材数、対象一覧、出典、調査メモ |

本機能の振る舞いについて本文書と `docs/preset.md` が異なる場合は、本文書を優先する。素材名や必要数が異なる場合は、推測でどちらかを採用せず、ゲーム内表示または信頼できる最新情報で再確認する。

## 2. 文書運用ルール

本機能の仕様を複数ファイルへ重複記載しない。新しい判断は本文書へ集約し、他の文書には本文書へのリンクだけを追加する。

以下を変更する作業では、コードと同じ作業内で本文書も更新する。

- プリセットの実装優先順位
- 最終ゴール、中継点、開始地点の意味
- 必要素材や不足数の計算方法
- プリセットID、バージョン、互換性
- 選択項目、分岐、対象別の置換方法
- API、DB、画面の責務
- 利用可能化の条件
- 未決事項の決定

文書更新時は次を守る。

1. 「決定事項」と「未決事項」を混在させない。
2. 未検証の数値は、検証済みと読める表現にしない。
3. 仕様変更時は末尾の変更履歴へ追記する。
4. 実装が仕様へ追従していない場合は、「現在の実装との差分」を更新する。
5. 数値だけを変更する場合も、影響するプリセットバージョンを判断する。
6. 本文書を更新せず、コードだけで新しい仕様を確定しない。

## 3. 目的

ユーザーが強化対象と現在地点を選ぶだけで、最終ゴールまたは指定した中継点までに必要な素材、現在所持数、不足数を確認できるようにする。

特に次を重視する。

- 最終ゴールまでの全体量を把握できること
- 直近の中継点までに必要な量へ表示を切り替えられること
- 同じ素材を複数段階で使う場合に正しく合算すること
- 属性、武器種、キャラ、作成ルートによる違いをプリセットで解決すること
- 一度入力した所持数を、同じユーザーの別目標でも再利用すること
- 未検証のゲーム情報を利用可能なプリセットとして公開しないこと

## 4. 対象外

初期実装では以下を行わない。

- ゲームアカウントからの所持数自動取得
- ゲーム内強化状況の自動取得
- 素材の自動消費・予約・目標間の割り当て
- 周回数やドロップ期待値の自動計算
- 外部攻略サイトの常時スクレイピング
- 十賢者の限界超越など、必要データが揃っていない後発段階
- 未検証データの暫定公開

## 5. 実装優先順位

プリセットは次の順で実装・利用可能化する。

1. 十賢者
2. 十天衆
3. 終末武器
4. ドラゴニックウェポン
5. レヴァンス武器
6. オリジンクラス
7. 破壊武器
8. 極星器

この順番は `docs/preset.md` の「初期実装順」より優先する。

十賢者で「対象別置換・中継点・累積計算」を実装し、十天衆で「作成ルート分岐」を追加する。その後の武器系プリセットは同じ計算基盤を利用する。

優先順位を飛ばして後続プリセットを実装する場合でも、先行プリセットの利用可能化を偽装したり、空の必要素材を登録したりしない。

## 6. 用語

### 6.1 プリセット

終末武器、十天衆、十賢者など、強化工程と必要素材の定義。

### 6.2 対象

プリセット内で育成する具体的な対象。十賢者ならキャラクター、武器系なら属性または武器を指す。

### 6.3 中継点

加入、礎武器5凸、最終上限解放など、進捗を区切って確認できる工程上の到達点。

### 6.4 現在地点

ユーザーが完了済みの最後の中継点。計算では現在地点を含めず、その次の工程から必要数へ加える。

`未着手` は画面上の選択肢として扱い、内部的には `startingStageId = null` とする。

### 6.5 最終ゴール

登録した目標が最終的に目指す到達点。十賢者の初期仕様では「最終上限解放＋4アビ取得」とし、内部の最終中継点は `4アビ取得` とする。

### 6.6 表示地点

現在画面で必要素材を確認する終点。表示地点を変えても、登録済みの最終ゴールは変更しない。

表示地点はMVPでは画面内の一時状態とし、DBへ保存しない。画面を開いた直後は最終ゴールを表示する。

### 6.7 必要数・所持数・不足数

- 必要数: 現在地点の次から表示地点までに必要な合計
- 所持数: ユーザーが現在保有している素材数
- 不足数: `max(必要数 - 所持数, 0)`

所持数は目標別ではなくユーザー別に共有する。複数目標へ素材を割り当てたり予約したりはしないため、それぞれの目標は同じ現在所持数を参照する。

## 7. 基本操作

### 7.1 目標登録

ユーザーは次を選択する。

1. プリセット
2. 対象
3. プリセット固有の選択項目
4. 現在地点

最終ゴールはプリセットの既定値を自動設定する。十賢者では `4アビ取得` を既定の最終ゴールとする。

将来、最終ゴールを短縮できる要件が確定した場合に備えて `goalStageId` は保存を続けるが、十賢者の初期UIでは最終ゴール変更を必須機能にしない。

### 7.2 進捗表示

登録後は次を表示する。

- プリセット名と対象名
- 現在地点
- 最終ゴール
- 次の未完了中継点
- 最終ゴールまでの進捗
- 表示地点の切り替え
- 表示地点までの必要素材合計
- 素材ごとの所持数と不足数
- 素材以外の必要条件
- 中継点ごとの内訳と達成状態

### 7.3 表示地点の切り替え

表示地点には、現在地点より後かつ最終ゴール以前の中継点だけを表示する。

推奨UI:

```text
表示先
- 次の中継点
- 礎武器5凸
- 至賢の領域全解放
- 最終上限解放
- 4アビ取得（最終ゴール）
```

切り替えは表示上の計算範囲だけを変更し、次を変更しない。

- DBに保存した最終ゴール
- 現在地点
- 所持数
- 完了済み中継点

## 8. 計算ルール

### 8.1 対象段階

段階の表示順を配列の添字で表す。

```text
開始位置 = 現在地点の添字
終了位置 = 表示地点の添字
計算対象 = 開始位置より後、終了位置以下の段階
```

`未着手` の開始位置は `-1` とする。

表示地点が現在地点以前の場合は必要数を0とし、「この中継点は達成済み」と表示する。表示地点が最終ゴールより後になる入力は受け付けない。

### 8.2 素材の合算

各段階の必要素材を `itemKey` 単位で合算する。

```text
合計必要数[itemKey]
= 段階必要数 × 対象本数 × 分岐による倍率
```

同じ素材が複数段階に存在する場合は加算する。表示名が同じでも `itemKey` が異なる素材は統合しない。

### 8.3 本数

本数指定がある武器プリセットは1～10本の整数を受け付ける。対象本数に比例する素材だけを乗算し、アカウント単位で一度だけ必要な条件は乗算しない。

### 8.4 置換

`対応属性エレメント`、`対応武器エレメント` などの抽象名を、そのままユーザー所持品のキーとして保存しない。

プリセット解決時に対象情報から具体的な `itemKey` と表示名へ置換する。

例:

```text
対応属性エレメント + 火属性
-> itemKey: element:fire
-> itemName: 火晶のエレメント
```

実際の表示名とキーは、マスタ定義を確認して確定する。

### 8.5 素材以外の条件

次は素材一覧と分けて条件として表示する。

- クエストクリア
- Rank
- キャラクターLv、ジョブLv
- 前提ジョブや前提強化の達成
- 選択肢だけで完了を表す工程

条件は以下の既存種別を基本にする。

- `check`: 達成・未達成
- `shared-number`: ユーザー内で共有する数値
- `goal-number`: 対象目標だけで使う数値

通貨や数量を素材として扱うか条件として扱うかは、プリセット間で同じ方針に統一する。

### 8.6 段階完了

段階完了は必要素材・条件を満たした場合に登録できる。完了登録によって所持数を自動減算しない。

現在地点より前の段階は完了済みとして扱う。中継点を手動完了した場合、次の中継点と計算範囲を更新する。

## 9. 十賢者プリセット

### 9.1 ゴール

既定の最終ゴールは「最終上限解放＋4アビ取得」。内部的には最後の中継点 `fourth-ability` までを対象とする。

十賢者の限界超越は初期対象に含めない。

### 9.2 対象

- アラナン
- カイム
- ガイゼンボーガ
- ハーゼリーラ
- カッツェリーラ
- ニーア
- フラウ
- エスタリオラ
- ロベリア
- マリア・テレサ

### 9.3 中継点

初期の表示順は次とする。

| ID | 表示名 | 備考 |
| --- | --- | --- |
| `arcarum-summon-complete` | アーカルム召喚石完成 | 加入前工程の終点 |
| `recruited` | 加入 | 加入済みユーザーの開始地点候補 |
| `foundation-weapon-5` | 礎武器5凸 | 交換から5凸までの工程を含む |
| `domain-complete` | 至賢の領域全解放 | 対象別素材を解決する |
| `final-uncap` | 最終上限解放 | 最終素材と前提条件を含む |
| `level-100` | Lv100 | 数値またはチェック条件として扱う |
| `fourth-ability` | 4アビ取得 | 初期仕様の最終中継点 |

`未着手` は中継点データを持たず、`startingStageId = null` で表現する。

### 9.4 初期表示

- 登録直後の表示地点: `fourth-ability`
- 次の中継点: 現在地点の直後にある未完了中継点
- 概要: 最終ゴールまでの必要数、所持数、不足数
- 詳細: 選択した表示地点までの合計と段階別内訳

### 9.5 必要な対象別データ

利用可能化する前に、10賢者それぞれについて次を定義する。

- 対応属性
- 対応召喚石
- 対応礎武器
- ウェリタス
- アストラ
- イデア
- フラグメント
- ブライト
- ヴェルム文書
- ヘイズ
- ゴスペル
- 六竜素材など対象で変化する素材
- 召喚石作成から加入までの段階別素材
- 礎武器の交換から5凸までの段階別素材

`docs/preset.md` に合計しかない工程は、段階別表示に必要な内訳を確認してから利用可能化する。

### 9.6 進行順と依存関係

画面上は9.3の順で表示する。ただし、礎武器と至賢の領域の正確な依存関係はゲーム内仕様を確認してから確定する。

単純な表示順だけで完了可否を決めず、必要な場合は段階へ `dependsOn` を定義できる設計にする。依存関係が未確認の間は推測による自動完了判定を実装しない。

## 12. プリセットバージョン

`ProgressGoal.presetVersion` を実際の定義解決に使用する。

- 公開済みのプリセット定義を同じversionのまま破壊的に変更しない
- 工程、必要数、条件、置換ロジックを変更する場合はversionを上げる
- 既存目標は原則として登録時versionを使い続ける
- 新versionへ更新する場合は、ユーザーが内容を確認できる導線を将来用意する
- 誤字修正など計算結果へ影響しない変更はversionを維持してよい

定義をコードで保持する場合も、`presetId + version` で過去定義を解決できる構造にする。

## 13. データ構造の方針

プリセット定義は最低限、次を表現できるようにする。

```ts
type ProgressPresetDefinition = {
  id: string;
  version: number;
  name: string;
  isAvailable: boolean;
  defaultGoalStageId: string;
  targets: ProgressPresetTarget[];
  fields: ProgressPresetField[];
  stages: ProgressPresetStage[];
};
```

`fields` は複数の選択項目を扱う。

- select: 属性、キャラ、武器、覚醒タイプ、作成ルート
- integer: 本数、目標Lvなど
- boolean: 任意工程を含めるか

段階の必要素材は、固定配列だけでなく対象・選択値を受け取って解決できるようにする。

DBの `selection` JSONは複数フィールドの保存に利用できる。MVPでは現在のDB項目を優先して再利用し、永続化要件を満たせない場合だけmigrationを追加する。

## 14. APIとサーバーの責務

サーバーを計算結果の正とする。

サーバーは次を行う。

- プリセット、対象、選択値、段階の検証
- 旧IDのalias解決
- `presetId + version` による定義解決
- 対象別素材の置換
- 本数と分岐倍率の適用
- 現在地点から表示地点までの素材合算
- 所持数と不足数の計算
- 素材以外の条件判定
- ユーザー所有者チェック

フロントエンドだけで必要数を確定しない。フロントエンドはサーバーが返した計算結果を表示し、表示地点の変更を問い合わせる。

表示地点をAPIへ渡す方法は、既存目標取得へのquery parameterを第一候補とする。

```text
GET /api/progress-goals/:id?displayStageId=final-uncap
```

APIの最終形式は実装前に既存クライアントとの整合を確認して確定する。

## 15. UI・レスポンシブ要件

`AGENTS.md` のCSS・レスポンシブ規則を適用する。

- 360px、768px、1280px以上で確認する
- スマートフォンでも目標登録、表示地点変更、所持数入力、段階完了を可能にする
- 素材表をページ全体からはみ出させない
- スマートフォンでは表をカード表示へ切り替えるか、明示した領域内スクロールにする
- 主要操作のタッチ領域は可能な限り約44pxを確保する
- 長い素材名、対象名、注意文を折り返す
- 色だけで不足・達成を区別しない
- 所持数入力はラベルと単位を明示する
- 表示地点を変更しても入力途中の所持数を失わない

## 16. 利用可能化の条件

プリセットを `isAvailable: true` にするには、以下をすべて満たす。

1. 対象一覧が揃っている
2. 対象別の置換表が揃っている
3. 利用可能な全段階の必要数が確認済み
4. 分岐と任意工程が定義済み
5. 計算テストがある、または代表ケースを再現可能な形で検証済み
6. 開始地点、表示地点、最終ゴールの範囲計算を確認済み
7. 360px、768px、1280px以上の表示を確認済み
8. `npm run typecheck` と `npm run build` が成功している
9. 本文書と `docs/preset.md` が実装内容と一致している

攻略サイト間で数値が異なる場合、素材名を確定できない場合、実装日付近にゲーム更新がある場合は利用可能化しない。

## 17. 十賢者の受け入れ条件

最低限、次のシナリオを確認する。

1. 未着手から4アビ取得までの必要素材を集計できる
2. 加入済みから4アビ取得までの必要素材を集計できる
3. 加入済みから最終上限解放までへ表示を切り替えられる
4. 表示地点を切り替えても登録済み最終ゴールが変わらない
5. 同じ素材が複数段階にある場合、必要数が合算される
6. 所持数を入力すると不足数が更新される
7. 所持数が必要数を超えても不足数が負にならない
8. 対象賢者を変えると対応素材が正しく変わる
9. 完了済み中継点の素材が計算対象へ戻らない
10. 旧プリセットID・versionの既存目標を読み込める
11. 他ユーザーの目標と所持数を更新できない
12. スマートフォンで主要操作と結果確認ができる

## 18. 現在の実装との差分

現行実装には以下の制約があるため、素材データ投入前に計算基盤を拡張する。

- プリセット固有の選択項目を1種類しか表示できない
- 本数と倍率を扱えない
- 抽象素材名の対象別置換がない
- 現在地点から表示地点までの累積集計がない
- 最終ゴールと表示地点が分離されていない
- 選択した最終ゴールより後の段階も表示・集計対象になり得る
- `presetVersion` を定義解決に使用していない
- 旧IDのaliasがない
- 段階間の依存関係を表現できない

空の `requirements` のまま `isAvailable: true` にしてはならない。

## 19. 未決事項

次は実装前または該当データ投入前に確定する。

- 十賢者の礎武器と至賢の領域の正確な依存関係
- 召喚石作成から加入までを何段階に分割するか
- 礎武器の交換、4凸、5凸を画面上でも分割するか
- `Lv100` を独立中継点とするか、4アビ取得条件としてだけ表示するか
- ルピなどの通貨を素材と条件のどちらへ統一するか
- プリセット新versionへのユーザー操作による更新方法
- 表示地点をURL queryへ保持するか、画面stateだけにするか
- 複数目標を横断した素材不足の参考合計を将来提供するか

未決事項を実装者の判断だけで確定しない。判断が必要になった時点で、選択肢と影響を整理して確認する。
