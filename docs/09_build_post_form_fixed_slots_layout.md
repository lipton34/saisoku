# Codex指示書：投稿フォームUIの固定スロット化・レイアウト崩れ修正

AGENTS.md の指示に従ってください。

今回の作業対象は、高難易度向け編成共有機能の投稿フォームUIの追加修正です。

前回、`docs/08_build_post_form_ui_rework.md` に基づいて投稿フォームUIを再修正しました。  
UI自体は改善されていることを確認できましたが、実画面確認の結果、以下の問題が残っています。

- 動的にUIが変化するとレイアウトが崩れる
- キャラ選択画面で「メンバー枠を追加」していくUIになっている
- グラブルの編成枠は基本的に決まっているため、初期状態から必要な枠を表示したい
- 武器・召喚石も同様に、追加式ではなく固定スロット式にしたい
- 候補一覧が多い場合にレイアウトが縦に伸びすぎる、または崩れる可能性がある

今回の修正では、キャラ・武器・召喚石を「追加して増やすUI」ではなく、基本的に「決まった枠を最初から表示するUI」に変更してください。

---

## 1. レイアウト崩れの修正方針

動的にUIが変化しても、ページ全体のレイアウトが崩れないようにしてください。

### 必須方針

- ページ全体に横スクロールを出さない
- キャラ枠・武器枠・召喚石枠のボード内だけ横スクロールを許可する
- 候補一覧は一度に表示しすぎない
- 候補一覧は2列表示を基本とする
- 候補が多い場合はページネーション形式にする
- 一度に表示する候補は最大10件までにする
- 検索時、ページ切り替え時、枠数切り替え時に親要素の幅が破綻しないようにする

### 横スクロールの扱い

以下の方針にしてください。

```text
ページ全体：横スクロール禁止
キャラ枠・武器枠・召喚石枠のボード内：横スクロール可
候補一覧：2列表示 + 最大10件 + ページネーション
```

CSSでは、ページ全体ではなく、編成ボード側にのみ `overflow-x: auto` を付けてください。

例：

```css
.formation-board-scroll {
  overflow-x: auto;
  max-width: 100%;
}

.formation-slot-grid {
  min-width: max-content;
}
```

ただし、スマホで操作しづらくなる場合は、スロット幅を適切に調整してください。

## 2. キャラ選択UIの修正

現在、キャラ選択画面ではメンバー枠を追加ボタンで増やすUIになっています。  
これをやめて、基本枠を最初から固定表示してください。

### 枠構成

キャラ枠は以下にしてください。

```text
主人公：1枠固定
フロントキャラ：3枠固定
サブキャラ：2枠 または 5枠を選択
```

フロント4枠という考え方ではなく、グラブル本家に合わせて以下の構成にしてください。

```text
主人公 + フロントキャラ3人
```

### フロント

- 主人公枠は固定表示
- フロントキャラは3枠固定表示
- 未選択を許可する
- 3キャラ未満の編成でも保存できるようにする
- フロントキャラの「追加」ボタンは不要

### サブ

サブは以下の切り替え式にしてください。

```text
サブ2枠
サブ5枠
```

サブ5枠は、ヴェルサシアなど一部クエストで必要になる想定です。

表示上は以下で構いません。

```text
サブ1
サブ2
サブ3
サブ4
サブ5
```

### サブ枠の注意点

- サブ2枠を選択した場合は、サブ1〜2を表示
- サブ5枠を選択した場合は、サブ1〜5を表示
- サブ枠が増えてもページ全体を横スクロールさせない
- はみ出す場合は、キャラ編成ボード内だけ横スクロールさせる
- 未選択枠を許可する
- サブキャラ追加ボタンは不要

## 3. 武器選択UIの修正

現在の武器UIは、枠追加で増やす形式になっている場合があります。  
これを、枠数選択式に変更してください。

### 枠構成

武器は以下の切り替え式にしてください。

```text
10枠
13枠
```

### 修正方針

- 初期表示は10枠
- ユーザーが13枠を選択すると13枠表示
- 10枠 / 13枠を選択できるUIを用意する
- 武器枠追加ボタンは廃止する
- 未選択枠を許可する
- 選択済み武器の差し替え・削除は維持する
- 重要度、本数、用途、代用など既存入力項目は維持する

### 表示例

```text
[武器枠数]
(●) 10枠
( ) 13枠

[Main Weapon]
  [メイン武器スロット]

[Weapon Grid]
  [武器1] [武器2] [武器3] [武器4] [武器5]
  [武器6] [武器7] [武器8] [武器9] [武器10]
```

13枠選択時は、武器11〜13を追加表示してください。

### レイアウト

- 武器ボード内のみ横スクロール可
- ページ全体には横スクロールを出さない
- スマホではスロットが潰れすぎないようにする
- 13枠表示時にフォーム全体が崩れないようにする

---

## 4. 召喚石選択UIの修正

召喚石も枠追加式ではなく、最初から固定枠を表示してください。

### 枠構成

召喚石は以下の構成にしてください。

```text
メイン召喚石：1枠固定
フレンド召喚石候補：1枠固定
サブ召喚石：6枠固定
```

### 修正方針

- メイン召喚石枠を固定表示
- フレンド召喚石候補枠を固定表示
- サブ召喚石は6枠固定表示
- 召喚石枠追加ボタンは廃止する
- 未選択枠を許可する
- 選択済み召喚石の差し替え・削除は維持する
- 重要度など既存入力項目は維持する

### 表示例

```text
[Main Summon]
  [メイン召喚石]

[Support Summon]
  [フレンド召喚石候補]

[Sub Summon]
  [サブ1] [サブ2] [サブ3] [サブ4] [サブ5] [サブ6]
```

### レイアウト

- 召喚石ボード内のみ横スクロール可
- ページ全体には横スクロールを出さない
- サブ6枠表示時にフォーム全体が崩れないようにする

## 5. 候補一覧のページネーション化

キャラ・武器・召喚石の候補一覧は、表示件数が多いとレイアウトが崩れやすいため、ページネーション形式にしてください。

### 共通要件

- 候補一覧は2列表示を基本とする
- 一度に表示する件数は最大10件
- 10件を超える場合はページネーションを表示する
- 検索結果に対してページネーションする
- 検索条件が変わったら1ページ目に戻す
- 0件の場合は「候補が見つかりません」などを表示する
- 0件時でも自由入力導線を表示する

### ページネーション例

```text
[前へ] 1 / 4 [次へ]
```

または、

```text
[前へ] [1] [2] [3] [4] [次へ]
```

実装が簡単な方で構いません。

### 注意点

- 候補一覧の高さが検索結果数に応じて大きく変わりすぎないようにする
- ページ切り替え時に親レイアウトが崩れないようにする
- キャラ・武器・召喚石で共通ロジックにできる場合は共通化してよい
- ただし大規模リファクタリングは不要

## 6. 未選択枠の扱い

キャラ・武器・召喚石は、枠が固定表示されていても未選択を許可してください。

### 必須要件

- 未選択枠があっても保存できる
- 未選択枠は「未選択」または「＋」で表示する
- 未選択枠をクリックすると、その枠が選択対象になる
- 候補一覧から選ぶと、その未選択枠に反映される
- 選択済み枠を削除すると未選択状態に戻る
- 未選択枠があることでバリデーションエラーにならないようにする

ただし、既存要件上必須の基本情報は従来どおりバリデーションしてください。

---

## 7. 既存機能を壊さないこと

今回の修正では、見た目とUI操作の安定化を中心に行ってください。

壊してはいけないもの：

- 投稿作成処理
- 投稿編集処理がある場合は編集処理
- プリセット反映処理
- キャラ・武器・召喚石の保存形式
- 参考URLの保存形式
- 投稿一覧表示
- 投稿詳細表示
- 認証・JWT + cookieログイン
- 既存のAPIレスポンス形式
- Prisma schema

必要がなければ Prisma schema は変更しないでください。

保存形式を変える必要がある場合は、既存表示や既存投稿に影響が出ないように最小限にしてください。

## 注意

今回の修正で最も重要なのは、以下です。

- 枠追加式UIをやめ、グラブルの編成画面に近い固定枠UIにする
- 未選択枠を許可し、3キャラ編成や未入力枠がある状態でも扱えるようにする
- 動的変化によるレイアウト崩れを抑える
- ページ全体には横スクロールを出さない
- 横スクロールが必要な場合は、キャラ・武器・召喚石の編成ボード内だけに限定する
- 候補一覧は2列・最大10件・ページネーションにする
- 保存形式やAPI、Prisma schema を不用意に変更しない
