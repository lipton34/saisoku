# Codex指示書：投稿フォームUIクリック遅延の原因調査・修正

AGENTS.md の指示に従ってください。

今回の作業対象は、高難易度向け編成共有機能の投稿フォームUIです。

前回までに、以下の修正は完了しています。

- 投稿フォームのステップ化
- `content-grid` の投稿フォーム専用横幅調整
- キャラ・武器・召喚石の編成スロット化
- キャラ：主人公1枠、フロント3枠、サブ2枠 / 5枠
- 武器：10枠 / 13枠
- 召喚石：メイン1枠、フレンド1枠、サブ6枠
- 候補一覧：2列、最大10件、ページネーション
- 候補選択ハンドラの `useCallback` 化
- 候補配列の `useMemo` 化
- 一部コンポーネントの `React.memo` 化
- サムネイル画像の `width` / `height` 指定
- `npm run typecheck` 成功
- `npm run build` 成功

しかし、実ブラウザで確認したところ、キャラ・武器・召喚石の候補をクリックした際に、選択結果がスロットへ表示されるまでの体感遅延がまだ残っています。

今回は、インフラ面ではなく、フロントエンド側のコード・レンダリング・state更新・CSS描画負荷を中心に、遅延原因を調査して修正してください。

---

## 目的

キャラ・武器・召喚石の候補をクリックしたときに、選択中スロットへ即時反映されるようにすることが目的です。

今回の目的は、UI仕様変更ではありません。  
既存の見た目・固定スロット仕様・保存形式は維持したまま、クリック時の体感レスポンスを改善してください。

---

## 対象ファイル

主に以下を確認してください。

- `src/components/BuildFormSteps.tsx`
- 投稿フォーム関連CSS
- 候補一覧コンポーネント
- スロット表示コンポーネント
- 固定スロット正規化処理
- キャラ・武器・召喚石選択時の state 更新処理

必要に応じて、関連ファイルも確認してください。

---

# 1. クリック時に何が再レンダリングされているか調査する

まず、候補をクリックしたときに、どの範囲が再レンダリングされているか確認してください。

特に以下が不要に再レンダリングされていないか確認してください。

- `BuildFormSteps` 全体
- 現在表示していないステップ
- Step 1 基本情報
- Step 5 その他情報
- `selected-part-picker`
- `reference-list`
- textarea群
- プリセットモーダル関連
- キャラ候補一覧全体
- 武器候補一覧全体
- 召喚石候補一覧全体
- 固定スロット正規化処理

1つの候補をクリックしただけで、フォーム全体が再描画されている場合は、そこが遅延原因である可能性が高いです。

---

# 2. 非表示ステップまで描画していないか確認する

ステップ形式UIで、現在表示していない Step 1〜5 の中身まで常にレンダリングされていないか確認してください。

もし全ステップをDOMに残したまま表示切替している場合、候補クリック時に非表示ステップも含めて再レンダリングされ、遅くなる可能性があります。

## 修正方針

- 現在表示中のステップのみ描画する
- 非表示ステップの重いUIは描画しない
- 特に候補一覧、textarea群、参考URLリスト、Step 5 の選択済みパーツ一覧は非表示時に描画しない
- 入力済みデータは state に保持し、DOMをアンマウントしても消えないようにする
- ステップ切り替え時に入力内容が消えないことを確認する

---

# 3. 固定スロット正規化処理が頻繁に走っていないか確認する

固定スロット化後、キャラ・武器・召喚石の枠数を整えるための正規化処理があるはずです。

この処理が候補クリックのたびに不要に走っていないか確認してください。

## 確認項目

- 候補クリックのたびに正規化処理が走っていないか
- 正規化処理が新しい配列・オブジェクトを毎回作っていないか
- 内容が変わっていないのに `setState` していないか
- `useEffect` の依存配列が広すぎないか
- キャラ選択時に武器・召喚石まで正規化していないか
- 武器選択時にキャラ・召喚石まで正規化していないか
- 召喚石選択時にキャラ・武器まで正規化していないか

## 修正方針

- 内容が完全に同じなら `setState` しない
- 必要なカテゴリだけ正規化する
- 毎回新しいオブジェクトを作らない
- 正規化処理は、枠数切り替え時や初期ロード時など、本当に必要なタイミングに限定する
- 候補クリック時には、選択対象スロットの更新を最優先する

---

# 4. form state 全体更新が原因になっていないか確認する

現在、投稿フォーム全体を1つの大きな state オブジェクトで管理している場合、1スロット更新だけでもフォーム全体が再評価されている可能性があります。

## 確認項目

- 候補選択時に form 全体を spread していないか
- キャラ1枠更新で、武器・召喚石・参考URL・Step 5 情報まで新しい参照になっていないか
- props として巨大な form オブジェクトを子コンポーネントに渡していないか
- `React.memo` していても、props の参照が毎回変わって無効化されていないか
- 候補クリックのたびに Step 5 の選択済みパーツ一覧を再生成していないか

## 修正方針

- 変更対象以外の配列・オブジェクト参照は維持する
- 子コンポーネントには form 全体ではなく、必要な値だけ渡す
- キャラ更新時はキャラ配列だけを更新し、武器・召喚石・その他情報は再生成しない
- 武器更新時は武器配列だけを更新する
- 召喚石更新時は召喚石配列だけを更新する
- 大規模なstate分割は不要だが、明らかに全体更新が原因なら必要最小限で更新処理を見直す

---

# 5. key が不安定で remount が発生していないか確認する

React の `key` が不安定だと、候補選択のたびにスロットや候補カードが remount され、遅く見える可能性があります。

## 確認項目

- `key` に `JSON.stringify` した値を使っていないか
- `key` に選択内容の name を使っていないか
- `key` に検索結果順で変わりやすい値を使っていないか
- スロットの `key` が選択内容変更で変わっていないか
- 候補カードの `key` がページ切り替え以外でも変わっていないか
- `key` に index だけを使っていて、並び替えや検索で不安定になっていないか

## 修正方針

- スロットは `slotId` など固定の識別子を `key` にする
- 選択内容が変わってもスロットの `key` は変えない
- 候補カードは master id があればそれを使う
- master id がない場合も、安定した一意キーを使う

---

# 6. 候補クリック時に検索条件やページ番号が不要に変わっていないか確認する

候補を選択しただけで、検索条件・ページ番号・候補一覧が再初期化されていないか確認してください。

## 確認項目

- 候補選択時に `searchKeyword` を更新していないか
- 候補選択時に `currentPage` を 1 に戻していないか
- 候補選択時に candidate list を再作成していないか
- 候補選択時に全候補の `filter` / `map` / `sort` が走っていないか
- 検索条件が変わっていないのにページネーション後の配列が再計算されていないか

## 修正方針

- 検索条件が変わった時だけ `currentPage` を1ページ目に戻す
- 候補選択だけでは検索条件・ページ番号を変えない
- 候補配列の `filter` / `sort` は `useMemo` で、依存配列を最小限にする
- ページネーション後の候補配列も必要な依存だけで再計算する

---

# 7. Step 5 の派生データ生成が重くないか確認する

Step 5 では、Step 2〜4で選択したキャラ・武器・召喚石を「選択済みパーツ」として表示しています。

候補クリックのたびに、Step 5 用の選択済みパーツ一覧を重く再生成している場合、表示中ステップがキャラ・武器・召喚石でも遅延につながる可能性があります。

## 確認項目

- Step 5 が非表示でも `selectedParts` を毎回生成していないか
- キャラ候補クリック時に武器・召喚石・Step 5 派生データまで再計算していないか
- `selectedParts` の生成で filter / map / concat を多用していないか
- `selectedParts` が props として広く渡されていないか

## 修正方針

- Step 5 表示時だけ `selectedParts` を生成する
- または `useMemo` で必要な依存だけに限定する
- 非表示ステップの派生データ生成を避ける
- 候補クリック時に Step 5 のUIが再描画されないようにする

---

# 8. CSS描画負荷・レイアウト計算を確認する

クリック時の state 更新が速くても、CSSの描画負荷やレイアウト再計算で遅く見える場合があります。

## 確認項目

- `transition: all` を使っていないか
- `box-shadow` が多すぎないか
- `filter` / `backdrop-filter` を多用していないか
- `grid-template-columns` に複雑な `min-content` / `max-content` を多用していないか
- `overflow-x` と `min-width: max-content` の組み合わせで毎回レイアウト計算が重くなっていないか
- スロットや候補カードのサイズがクリックのたびに変わっていないか
- 選択状態によってカードサイズが変わっていないか

## 修正方針

- `transition: all` は使わない
- transition 対象は `border-color` / `background-color` / `transform` 程度に限定する
- スロットや候補カードの width / min-width / height を安定させる
- 選択状態でサイズが変わらないようにする
- 重い影やfilterは控えめにする
- 必要であれば `contain: layout paint;` を検討する

例：

```css
.candidate-panel {
  contain: layout paint;
}

.formation-board-scroll {
  contain: layout paint;
}

.formation-slot,
.candidate-card {
  transition: border-color 120ms ease, background-color 120ms ease, transform 120ms ease;
}
```

ただし、`contain` の指定で表示崩れが出る場合は無理に使わないでください。

---

# 9. 画像表示が「遅延に見える」原因になっていないか確認する

インフラ面が主原因ではない想定ですが、画像表示だけが遅れている場合、体感としてクリック反応が遅く見えることがあります。

## 確認項目

- 候補クリック後、名前は即時表示されているか
- 画像だけが遅れて表示されていないか
- スロット表示が画像読み込み完了を待っていないか
- サムネイルがない場合に破損画像で待っていないか
- placeholder が即時表示されるか

## 修正方針

- スロットには名前を即時表示する
- 画像読み込み完了を待ってスロット表示しない
- サムネイルがない場合は即時プレースホルダーを表示する
- img の `width` / `height` は維持する
- `loading="lazy"` は候補一覧では維持してよい
- 選択済みスロットの画像は、必要なら lazy を外すことも検討する

# 10. 一時的な計測ログで原因を切り分ける

必要であれば、一時的に計測ログを入れて原因を切り分けてください。

例：

```ts
console.time("selectCandidate");
// 選択処理
console.timeEnd("selectCandidate");
```

または、開発時のみレンダリング回数を確認するログを入れても構いません。

ただし、最終的な変更には不要な `console.log` / `console.time` を残さないでください。  
残す場合は、開発時のみ動作する条件付きにしてください。

---

# 優先して修正してほしい方向

原因が複数ある場合、以下の順で優先してください。

## 優先度1

- 現在表示中のステップだけレンダリングする
- 非表示ステップの候補一覧や入力欄を再レンダリングしない
- 選択対象以外のキャラ・武器・召喚石配列を再生成しない
- 固定スロット正規化で不要な `setState` をしない

## 優先度2

- スロット・候補カードの `key` を安定させる
- 候補クリックで検索条件やページ番号を不要に変更しない
- props に巨大な form オブジェクトを渡さない
- Step 5 の派生データ生成を非表示時に行わない

## 優先度3

- `transition: all` をやめる
- 重い `box-shadow` / `filter` / `backdrop-filter` を減らす
- スロットや候補カードのサイズを安定させる
- 選択済みスロットの画像表示を軽くする

---

# 変更してはいけないこと

以下は原則変更しないでください。

- UIの大きな見た目
- 固定スロット式の仕様
- キャラ：主人公1枠、フロント3枠、サブ2枠 / 5枠
- 武器：10枠 / 13枠
- 召喚石：メイン1枠、フレンド1枠、サブ6枠
- 候補一覧：2列、最大10件、ページネーション
- 投稿保存形式
- APIレスポンス形式
- Prisma schema
- 認証・JWT + cookieログイン
- プリセット反映仕様

今回の目的は、UI仕様の変更ではなく、クリック時の反応遅延の原因調査と改善です。
